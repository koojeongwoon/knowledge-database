# 프로젝트 규칙 (Rules)

## 1. 로컬 지식베이스 우선 참조 지침
- **모든 사용자의 질문 및 요청에 대해 답변을 작성하기 전에, 우선적으로 무조건 `search_wiki_knowledge` MCP 도구를 실행**하여 로컬 지식베이스의 누적 기록을 조회하십시오.
- 관련 지식이 없어 임계치 아래로 판단되더라도 일단 항상 1회 조회하는 것을 원칙으로 하며, 조회된 결과가 있는 경우 이를 최우선으로 반영하여 답변을 작성해야 합니다.

## 2. Context7 (기술 문서 RAG) MCP 사용 지침
- **사용 시점**:
  - 라이브러리, 프레임워크, SDK, API, CLI 도구, 클라우드 서비스(React, Next.js, Spring Boot, Prisma, Tailwind 등)의 최신 문서를 참조해야 할 때.
  - API 문법, 마이그레이션 가이드, 환경 설정 정보 등 학습 데이터만으로는 부정확할 수 있는 기술적 스펙을 확인할 때.
- **금지 시점**:
  - 코드 리팩토링, 스크립트 작성, 비즈니스 로직 디버깅, 코드 리뷰.
  - 일반적인 프로그래밍 개념(예: OOP, 자료구조 개념 등) 질문.

## 3. Playwright (브라우저 자동화/렌더링) MCP 사용 지침
- **사용 시점**:
  - JavaScript 렌더링이 필수적인 동적 웹 페이지(SPA)의 데이터를 수집해야 할 때.
  - 웹 화면을 캡처(Screenshot)하여 레이아웃, 스타일, UI 에러 상태 등을 시각적으로 분석해야 할 때.
  - 버튼 클릭, 로그인, 폼 작성 등 웹페이지 내 사용자 상호작용(Interaction)이 동반되는 정보를 수집해야 할 때.
- **금지 시점**:
  - JavaScript 실행이 불필요한 단순 정적 웹페이지, API 문서, 텍스트 정보의 경우 플레이라이트 대신 더 가볍고 빠른 `read_url_content` 등의 텍스트 기반 도구를 우선 사용하십시오.

## 4. inbox/ 대기 지식의 대화형 소화 (Knowledge Assimilation) 지침
- **동작 규칙**:
  - 에이전트는 대화 세션이 기동되면 가장 먼저 `inbox/` 디렉터리에 처리되지 않은 날것의 지식 파일들이 존재하는지 확인해야 합니다.
  - 감지된 파일이 있다면, 단순히 기계적으로 이관하는 대신 **"inbox/에 소화되지 않은 지식(N개)이 감지되었습니다. 이 문서들을 읽어본 결과 [대략적인 핵심 주제]에 대한 내용으로 보입니다. 지식베이스로 온전히 흡수하기 위해 다음을 진행해 드릴까요?"**라며 적극적으로 대화를 시작하십시오.
  - **대화 및 정제 과정**:
    1. 문서에 포함된 **외부 URL(블로그, 깃허브 등)이 있다면 RAG/웹 검색 도구를 활용하여 필요한 내용을 추가로 수집**해 콘텐츠를 풍부하게 살찌웁니다.
    2. 흩어진 파편적 문서들이 서로 연관되어 있다면, 하나로 합쳐 종합적인 맥락을 갖춘 완성형 지식 문서로 가공할 것을 제안하십시오.
    3. 필요하다면 관련 시각 자료(이미지 등)를 생성하거나 다운로드하여 `assets/` 디렉터리에 에셋으로 묶어 넣는 방안을 논의하십시오.
  - 사용자 조율을 통해 풍성해진 지식을 최종적으로 `qa/` 저널 및 `topics/` 토픽 문서로 확정 커밋(`commit_new_knowledge` 도구 사용)하십시오.

## 5. 지식화 시 세션 아티팩트(Artifacts) 이관 동의 확인 지침
- **모든 지식 영속화 및 커밋(`commit_new_knowledge` 사용)을 실행하기 전에**, 현재 대화 세션에서 작성된 아티팩트(설계서, 보고서 등)가 존재하는지 반드시 확인하십시오.
- 아티팩트가 존재할 경우, 사용자에게 **"이전에 작성된 아티팩트(예: [파일명](링크))도 함께 영구 지식베이스에 복사하고 링크를 연결해 드릴까요?"**라고 물어보고 승인을 얻은 뒤 프로세스를 진행해야 합니다.
- 사용자가 동의하면 아티팩트의 경로를 `commit_new_knowledge` 도구의 `resource_paths`에 추가하여 이관을 보장하십시오.
